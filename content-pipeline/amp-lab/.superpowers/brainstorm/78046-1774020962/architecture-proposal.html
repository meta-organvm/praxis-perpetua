Object Lessons — Full Site Architecture

Two approaches. Click your preference, then review the full design below.

A

Next.js + MDX

Ecosystem consistent. Matches stakeholder-portal, uses App Router + server components. React where you need interactivity (submission form, pipeline visualization, collaborator view). Heavier runtime but reuses ORGANVM's existing stack. Deploys to Vercel or Cloudflare Pages.

Strengths

  • Same stack as stakeholder-portal — shared knowledge
  • React components for interactive elements (pipeline viz, film tables, submission form)
  • API routes for form submission backend
  • Strong TypeScript support
  • ISR (Incremental Static Regeneration) for dynamic + static hybrid

Trade-offs

  • Ships more JS to the browser than a content site needs
  • MDX setup more complex than Astro's content collections
  • Heavier build toolchain
B

Astro + MDX (Recommended)

Purpose-built for content. Ships zero JS by default — only hydrates interactive islands (form, pipeline viz). Content collections with type-safe frontmatter. Fastest possible load for an editorial site. Uses React components only where needed via islands architecture.

Strengths

  • Zero JS by default — editorial pages are pure HTML + CSS
  • Content Collections: type-safe schemas for episodes, essays, objects, films
  • Islands architecture: React/Svelte only for interactive elements
  • Built-in RSS, sitemap, SEO support
  • Fastest editorial site possible — perfect for Sight & Sound aesthetic
  • Still uses React components where interactivity is needed

Trade-offs

  • Different framework from stakeholder-portal
  • Team learning curve (though Astro is simpler than Next.js)
  • API routes less mature than Next.js (use Cloudflare Functions or external)

Submission System

/submit — Viewer Contributions
Three submission types:
🎯 Suggest an Object
"You should cover umbrellas — they appear in Singin' in the Rain, Oldboy, Cherbourg..."
Fields: object name, why it matters, 3+ film examples
🎬 Flag a Film
"In Stalker (1979), there's a glass of milk on the table in the final scene..."
Fields: film, year, object, scene description, timestamp if known
📎 Submit a Clip
"Here's a timestamped YouTube link to the mirror scene in Black Swan..."
Fields: film, object, URL, timestamp, context note
Backend: Submissions → Cloudflare D1 (SQLite) or Neon Postgres. Reviewed in /collaborator dashboard. Approved submissions feed into the film database and credit the contributor.

Replicable Template Architecture

The Pattern — Fork for Future Research Units
The site is structured so a future research unit (e.g., "Gesture Lessons: Recurring Gestures in Theatre" or "Sound Objects: Recurring Sounds in Music") can fork the repo and swap in:
What Changes Per Unit
site.config.ts — name, tagline, domain, social links
theme.css — colors, typography adjustments
content/ — all content collections (episodes, essays, objects, films)
data/ — film/work databases, pipeline status
public/ — images, favicons, OG assets
What Stays the Same
Page layouts and components
Content collection schemas
Submission system
Collaborator dashboard
Pipeline visualization
Editorial design system (typography, spacing, components)
SEO infrastructure (sitemap, RSS, structured data, OG generation)
# Fork for a new research unit
gh repo create omni-dromenon-machina/gesture-lessons --template omni-dromenon-machina/object-lessons
# Edit site.config.ts, swap content/, deploy

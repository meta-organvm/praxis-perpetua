# SOP: Release -> Catch -> Handoff (RCH)

**Version:** 1.0.0 | **Date:** April 4, 2026 | **Status:** Active
**Scope:** The canonical cycle for transmuted abstract human intent into fortified, merged codebase reality.

---

## 1. Ontological Purpose

The RCH protocol prevents "premature engineering collapse." It ensures that abstract ideas are fully explored in their potential state (the Multiverse) before being frozen into bureaucratic plans and delegated to automated execution. It positions the primary AI agent as a **Conductor** who manages **Process Bits** and **GitHub Agents** (Jules, Copilot) to achieve maximum throughput with zero loss of systemic coherence.

---

## 2. Phase I: RELEASE (Human)

The human user "releases" a prompt. This is often abstract, visionary, or underspecified. 
- **Action:** The agent must *catch* this language without reducing it.
- **Rule:** Do not jump to the first solution.

---

## 3. Phase II: CATCH (Multiversing & Bureaucracy)

### 3.1. Multiversing
The agent must "spray" a variety of approaches.
- **Suggestion 1:** Incremental/Safe.
- **Suggestion 2:** Structural/Radical.
- **Suggestion 3:** Speculative/Infinite Potential.
- **Clarification:** Identify what is unclear. Do not simplify; *sharpen* the boundaries.

### 3.2. Hyper-Bureaucratic Planning
Once a path (or fusion) is selected, create exhaustive plans (`SOP--planning-and-roadmapping`).
- **Goal:** Plan so deeply that the engineering feels mechanical.
- **Artifact:** Versioned plan file in `<repo>/.claude/plans/`.

### 3.3. Task Registration
Log all tasks into the **Universal Agenda** (`INST-INDEX-RERUM-FACIENDARUM.md`).
- **Sync:** Tasks become GitHub issues on the project board.

---

## 4. Phase III: HANDOFF (Agent Execution)

### 4.1. Designing the Action
Assign specific "process bits" to GitHub Agents.
- **Jules:** Multi-file refactors, dependency upgrades, exhaustive test suites.
- **Copilot:** Surgical fixes, targeted logic, UI components.
- **Tool:** `start_new_jules_task` / `assign_copilot_to_issue`.

### 4.2. Fortification & Review
The agent (or a draft PR) returns a "completed process bit."
- **Action:** The Conductor reviews, runs tests, and *fortifies* the draft.
- **Validation:** Run `SOP--structural-integrity-audit`.

### 4.3. Merging
The fortified artifact is sent back to GitHub as a proper PR and merged into the main stream.

---

## 5. Starter Research Questions

- How many potential pathways exist for this prompt?
- Which pathway maximizes "infinite potential"?
- What specific "engineering bits" can be offloaded to Jules or Copilot?
- Is the IRF updated with the new trajectory?

---

## 6. Output Artifacts

- **Multiverse Spray:** A list of distinct suggestions.
- **Bureaucratic Plan:** Exhaustive task list.
- **IRF Entry:** Unique ID task in the registry.
- **PR:** The final manifestation of the fortified work.

---

## 7. Verification

- [ ] At least 3 distinct pathways suggested.
- [ ] Plan is detailed enough to feel "non-engineering."
- [ ] Tasks mapped to specific GitHub Agents.
- [ ] PR includes fortification (tests/docs).
